Going Live — The Feishu Bot

From terminal tool to team-wide chat integration

The agent works great in the terminal. You type a job URL, you get a polished analysis back. But what if your whole team wants to use it — without opening a laptop, without touching a command line? That's where Feishu comes in.

Feishu lets you build a bot that lives right inside a group chat. A teammate pastes a job link, the bot replies with a full analysis. No terminal, no setup — just a message. The magic that makes this possible is a webhook: Feishu delivers every incoming message to your server as an HTTP POST request.

The Flask server in app/channels/feishu_bot.py is the receptionist at the front desk — it greets every incoming webhook, hands off the real work to colleagues in the back office, and keeps the door open for the next visitor.

What is Feishu/Lark?
Feishu is ByteDance's enterprise collaboration platform, dominant in Chinese tech companies. Outside China it's called Lark. The same codebase, the same API. If you've used Slack or Teams, the mental model is identical — the only difference is the package name: lark_oapi.

The 3-Second Problem

Here's the catch. When Feishu sends a webhook to your server, it expects a response within 3 seconds. If your server takes longer, Feishu assumes something went wrong and retries — sending the same message again and again. Your agent calls Claude, parses a job description, writes a cover letter. That takes 15–20 seconds.

Do the maths: 15 seconds > 3 seconds. If you try to do all the work before replying, every single message triggers a cascade of retries. The user ends up with five identical analyses in their chat. Not great.

The Insight: Acknowledge First, Work Later
The solution is to split the response from the work. The Flask route returns {"msg": "ok"} with a 200 status the moment the webhook arrives — before any analysis happens. The heavy lifting is handed off to a background thread that starts simultaneously and sends the real reply whenever it's ready.

Think of it like a receptionist at a busy office. When a visitor arrives, the receptionist doesn't make them stand at the door while the whole office scrambles to prepare. She says "Welcome, I'll let the team know you're here" — and the visitor is seated immediately. The colleagues in the back get the message and do the actual work at their own pace.

The Webhook Handler, Decoded

Every line has a reason

CODE 

@app.route("/feishu/webhook", methods=["POST"])
def webhook():
    body = request.get_json(silent=True) or {}
    if body.get("type") == "url_verification":
        challenge = body.get("challenge", "")
        return jsonify({"challenge": challenge})
    event_type = body.get("header", {}).get("event_type", "")
    if event_type == "im.message.receive_v1":
        thread = threading.Thread(
            target=_process_message,
            args=(body,),
            daemon=True
        )
        thread.start()
    return jsonify({"msg": "ok"})
PLAIN ENGLISH

Line 1–2: Register this function as the handler for any POST request arriving at /feishu/webhook.

Line 3: Parse the incoming JSON body. If it's malformed, default to an empty dict instead of crashing.

Lines 4–6: Handle the URL verification challenge — a one-time setup handshake Feishu requires to confirm you own the server.

Line 7: Extract the event type from the nested header. Feishu wraps metadata in a header object.

Lines 8–13: If it's a new incoming message, spin up a daemon thread pointing at _process_message and start it immediately.

Line 14: Return 200 OK right away — regardless of how long the background thread takes. Feishu is satisfied and won't retry.

One Agent, Three Interfaces

The power of decoupled architecture

Here's something elegant hidden in this codebase. The CLI, the interactive REPL, and the Feishu bot are three completely different front doors — but they all walk into the same room. Every single interface ultimately calls one function: handle_user_request() in app/agent/orchestrator.py.

This is decoupled architecture in practice. The interfaces know nothing about Claude, cover letters, or JSON parsing. The orchestrator knows nothing about Feishu message IDs, terminal ANSI colours, or argparse arguments. They only share one contract: pass in a string, get back a summary string and an optional list of saved files.

Why This Pattern Matters
Want to add a Slack bot tomorrow? A web API? A mobile app? You write the interface adapter — roughly 50 lines — and the entire agent brain comes for free. The handle_user_request() function is your stable contract. Build around it, not inside it.

Key Takeaways from This Course
  • 01 Intent parsing decouples input from action. The system understands your intent before deciding what to do — no rigid command syntax required.
  • 02 Structured prompts produce structured outputs. Pydantic schemas enforce that Claude returns parseable, typed data — not freeform prose.
  • 03 Threading solves real-time constraints. Any time you integrate with a system that has a timeout, the acknowledge-then-process pattern is your friend.
  • 04 One core function, many interfaces. handle_user_request() is the system's stable contract. The CLI, REPL, and Feishu bot are just different front doors into the same room.
